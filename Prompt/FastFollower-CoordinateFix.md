# Fast-Follower Prompt — Floor-Plan Coordinate Fix

Act as a senior frontend engineer with strong CSS layout instincts.

The v1 desk-booking app renders coloured dots over PNG floor plans to represent desks. The dots are visibly misaligned with the desks drawn on the PNGs — see `test/Screenshot 2026-05-11 155129.png` and `test/Screenshot 2026-05-11 155146.png`. Fix the misalignment so every desk dot sits inside the desk it represents.

## Objective

Make desk markers visually align with the drawn desks on `floorplans/ground.png` and `floorplans/first.png` across all common viewport widths, without breaking the existing booking flows.

## Where the bug lives

- **Coords are stored normalised 0..1** in `data/desks.json` (`x` and `y` per desk).
- **They are rendered as CSS percentages** in `client/src/components/FloorPlan.jsx`:
  ```jsx
  style={{ left: `${d.x * 100}%`, top: `${d.y * 100}%` }}
  // marker is then centred with `transform: translate(-50%, -50%)` in styles.css
  ```
- **The container** `.floor-wrap` in `client/src/styles.css` is `width: 100%` with **no fixed aspect-ratio**. The image inside has `width: 100%; height: auto`, so the container's height tracks the PNG's intrinsic aspect ratio — but markers' `top: y * 100%` is computed against the container's *layout* height, which can drift if the surrounding grid forces a different height than the natural image height.
- **Coords are generated by** `server/scripts/generate-desks.js` (`DX=0.035`, `DY=0.045`, `POD_GAP_X=0.03`, six zone clusters per floor). The pod-grid maths assumes the PNG drawn-desk zones sit at specific x/y windows; if the PNG was authored with different proportions, every dot shifts in the same direction.
- **No override file currently exists** on `main` (a `data/desk-positions.json` lives on the `ben_test_v1` branch but has not been merged).

## In Scope (Build These)

1. **Measure the PNGs.** Record native pixel dimensions of `floorplans/ground.png` and `floorplans/first.png`. Commit the values into a short comment block at the top of `generate-desks.js` so future contributors can sanity-check the grid maths.

2. **Pin the floor-plan aspect ratio.** Add a per-floor CSS `aspect-ratio` on `.floor-wrap` so the container height matches the PNG's native aspect ratio exactly. Use a data attribute on the wrapper (e.g. `data-floor="ground"`) and CSS rules:
   ```css
   .floor-wrap[data-floor="ground"] { aspect-ratio: <w> / <h>; }
   .floor-wrap[data-floor="first"]  { aspect-ratio: <w> / <h>; }
   ```
   Update `FloorPlan.jsx` to pass the floor prop through as `data-floor`.

3. **Add an admin calibration overlay.** When `?calibrate=1` is in the URL, render:
   - A subtle 10% grid over the floor plan.
   - Live `x`, `y` (3-decimal) tracking the mouse position inside the floor-wrap.
   - A click handler that writes the clicked desk's id + the new `x/y` to an in-memory queue and exposes a "Copy JSON" button that emits the override map.
   The overlay must be off by default and have zero impact on the normal booking experience.

4. **Per-desk pixel overrides.** Introduce `data/desk-positions.json` of the shape:
   ```json
   {
     "G-001": { "x": 0.061, "y": 0.094 },
     "G-002": { "x": 0.096, "y": 0.094 }
   }
   ```
   The server merges these overrides on top of `desks.json` coords when serving `GET /api/desks`. Document the precedence: override file > generated coords.

5. **Re-generate or hand-tune.** Either:
   - Adjust the constants and zone anchors in `generate-desks.js` until alignment is correct without overrides; or
   - Generate a starter `desk-positions.json` from the calibration tool and store deltas there.
   Either approach is acceptable provided the resulting build looks correct.

6. **Capture before/after screenshots.** Save new screenshots into `test/` named `Screenshot-coord-after-<floor>.png` for both floors. Keep the originals in place.

## Out of Scope

- Redesigning the floor-plan PNGs themselves.
- Adding new desks, zones, or attributes.
- Re-themeing the marker visuals beyond what's needed to verify alignment (e.g. you may temporarily enlarge markers during calibration but must restore the production size before commit).
- Changes to booking, suggestions, auto-release, or any non-rendering logic.

## Delivery Instructions

- Work in small, verifiable steps — start by adding the aspect-ratio fix and confirm a visible improvement before touching coords.
- Keep `data/desks.json` as the source of truth; treat `desk-positions.json` as a thin overlay.
- Verify on viewport widths 1440px, 1024px, and the ≤900px single-column breakpoint declared in `styles.css`.
- Don't regress the marker hover/tooltip behaviour in `DeskTooltip` (positioned with the same percentage maths).

## Success Criteria

- Every visible desk dot on both floor plans sits inside its drawn desk on the PNG, at 1440px, 1024px, and ≤900px widths.
- The calibration overlay is reachable via `?calibrate=1`, invisible without it, and produces JSON copy-able into `desk-positions.json`.
- No regressions in booking, check-in, release, suggestions, or `/manage`.
- Before/after screenshots committed in `test/`.
